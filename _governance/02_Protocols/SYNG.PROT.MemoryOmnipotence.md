## I. Purpose: The Fivefold Memory Architecture

The **Memory Omnipotence Protocol** establishes a unified, multi-layered cognitive repository for the Synarchy. It
mandates the transition from ephemeral session logs to a persistent, hierarchical memory graph, ensuring absolute recall
of user-validated "Gems" and architectural wisdom.

---

## II. The L1-L5 Hierarchy

| Layer             | Type                | Definition                                                                               |
| :---------------- | :------------------ | :--------------------------------------------------------------------------------------- |
| **L1: GEMS**      | **The Muse**        | High-priority facts confirmed by the USER. Managed in `GEMS.md`.                         |
| **L2: KINETIC**   | **Active Session**  | Current session context, tool results, and temporary state. Stored in `axion_memory.db`. |
| **L3: SEMANTIC**  | **Knowledge Graph** | Vectorized RAG index of all core project files (`nova_forge`, `axion-core`).             |
| **L4: SOVEREIGN** | **Architectural**   | Immutable design laws and "Self" identity stored in `LEARNINGS.md`.                      |
| **L5: META**      | **Evolutionary**    | Performance logs, error patterns, and self-optimization data in `ERRORS.md`.             |

---

## III. The Gem Extraction Cycle (The Muse)

1. **Identification**: Detect high-impact facts or user preferences during conversation.
2. **Validation**: Confirm with the user if the insight should be "Gemified."
3. **Canonization**: Store in the L1 `memory_gems` table with high activation score.
4. **Recall**: L1 Gems are prioritize in every `retrieve_memories` call.

---

## IV. Sovereign Bridge Protocol (L2 -> L4)

- **Threshold**: Memories with a Coherence score > 0.9 and 3+ reactivations are candidates for L4.
- **Transmutation**: The `@memory-specialist` shard formalizes the memory into a markdown entry in
  `.learnings/LEARNINGS.md`.
