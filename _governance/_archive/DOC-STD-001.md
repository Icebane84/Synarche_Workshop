## **1.0 Operational Mandate**

**What:** This document establishes the non-negotiable syntax, structure, and documentation rules for all software
generated by the Synarche. **How:** It is enforced via "Preventative QA" (Axion-Core) and manual review. Every line of
code must pass these standards before being committed to the Master Branch. **Why:** To ensure "Functional Coherence."

refactor the codebase recursively without losing context.

## **2.0 The Relational Naming Convention (RNC)**

_This section defines how entities are named to ensure instant recognition of their scope and function._

### **2.1 General RNC Rules**

- **No Magic Numbers:** All operational constants must be named variables (e.g., `MAX_RETRY_ATTEMPTS = 3` instead of
  just `3`).
- **Boolean Predicates:** Boolean variables must answer a question (e.g., `is_active`, `has_permission`, `can_execute`).

### **2.2 Language-Specific RNC**

| Entity Type | Python Style | TypeScript Style | Example | | **Variables** | `snake_case` | `camelCase` | `user_id` /

## **3.0 Language Protocols**

### **3.1 Python (The Backend Logic)**

- **Type Hinting:** **MANDATORY.** All function signatures must utilize the `typing` module (`List`, `Dict`, `Optional`,
  `Any`).
  - _Code:_ `def fetch_data(id: int) -> Optional[Dict[str, Any]]:`
- **Docstrings:** Must follow the **Google Style Guide**. Must be present for every Class and Public Method.
- **Imports:** Sorted alphabetically. Standard library first, third-party second, local application third.

### **3.2 TypeScript (The Interface & Architecture)**

- **Strict Mode:** The `tsconfig.json` must have `"strict": true`.
- **Explicit Any:** usage of `any` is **FORBIDDEN** unless explicitly documented with a `// TODO: Fix implicit any`
  comment. Use `unknown` or defining a Generic `<T>` instead.
- **Interfaces vs Types:** Use `interface` for defining public API contracts (e.g., Data Models). Use `type` for unions
  and intersections.

## **4.0 Documentation & Knowledge Graph Integration**

### **4.1 The "Why" Rule**

Comments must explain the _intent_, not the _mechanism_.

- _Forbidden:_ `x = x + 1 # Increment x`
- _Required:_ `retry_count += 1 # Increment to track connection attempts against MAX_RETRY limit`

### **4.2 Artifact Linking**

When code implements a specific blueprint or governance document, it must reference the Artifact ID in the file header.

```python

# Reference: UMB-IDE-EXT-001

# Context: Implements the parser logic for the Axion Core
```

## **5.0 Version Control & Commit Standards**

### **5.1 Commit Messages**

We utilize **Conventional Commits** to ensure the **AISTF** can parse our history.

- Structure: `<type>(<scope>): <subject>`

- Types:
  - `feat`: A new feature
  - `fix`: A bug fix
  - `docs`: Documentation only changes
  - `refactor`: A code change that neither fixes a bug nor adds a feature
  - `chore`: Maintenance (build tasks, package manager configs)

- _Example:_ `feat(parser): implement UMB JSON schema validation`
