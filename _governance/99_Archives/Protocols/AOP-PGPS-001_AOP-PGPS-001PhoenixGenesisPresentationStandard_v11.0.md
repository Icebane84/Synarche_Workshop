## IV. Operationalizing the Standard: Core Formatting Directives

### 4.1. Macro-Structural Hierarchy

The Phoenix Genesis Presentation Standard mandates a precise macro-structural hierarchy for all documents. The primary
title of an artifact must exclusively utilize a single hash symbol (e.g., `# H1`), signifying the highest level of
organization. Major sections within a document are designated with double hash symbols (e.g., `## H2`), while
sub-sections are structured using triple hash symbols (e.g., `### H3`). For consistent rendering and improved
readability, a single space must follow the hash symbols, and a single blank line must precede and follow each
heading. Trailing hash symbols are strictly prohibited to maintain consistency and clarity.

To clearly delineate major, distinct sections within a document, a horizontal rule consisting of exactly three
hyphens (`---`) is mandated. This horizontal rule must start at the absolute left margin (zero indentation) and be
surrounded by a single blank line. The use of other markers, such as asterisks or underscores, for horizontal rules
is strictly prohibited to avoid ambiguity. This consistent application of structural elements ensures that headings are
reliably recognized as top-level components, preventing misinterpretation and facilitating accurate parsing for
features like Table of Contents generation and hierarchical navigation by AI.

### 4.2. Textual Organization & Emphasis

The PGPS defines precise rules for the organization and emphasis of textual content to ensure clarity and consistency.
All paragraphs must commence at the absolute left margin, with zero indentation, and be separated from other elements
by a single blank line. Indenting the first line of a paragraph is strictly prohibited unless it is an explicit part
of a list item, as this can lead to misinterpretation as code blocks or list items by Markdown parsers.

For explicit line breaks within a paragraph (soft wraps, functionally equivalent to HTML `<br />`), precisely two
trailing spaces must be used at the end of the line. For the initiation of new paragraphs, a single blank line is
sufficient. This distinction is crucial for AI to differentiate between a soft line break, which indicates a
continuation of a thought within the same semantic unit, and a hard paragraph break, which signifies the start of a
new, distinct semantic unit.

Emphasis within text is achieved using asterisks: single asterisks (`*`) are used for italics, and double asterisks
(`**`) are used for bold text. The use of underscores (`_`, `__`) for emphasis is strictly avoided to prevent
potential conflicts with filenames or other syntax elements, particularly in technical documents. Blockquotes must be
consistently prefixed on every line with a single greater-than symbol (`>`) followed by a single space, including any
blank lines that serve to separate paragraphs within the blockquote. For nested blockquotes, each successive level
requires an additional `>` (e.g., `>>` for the second level). This consistent application ensures the entire
blockquote is recognized as a single, distinct semantic unit, crucial for AI to correctly attribute quoted content or
isolate specific sections for analysis.

### 4.3. List Formatting: The Hierarchical Knowledge Weave

The PGPS establishes rigorous rules for list formatting to ensure a clear hierarchical knowledge weave within
documents. Bulleted list items must consistently begin with a hyphen (`-`) followed by a single space. The use of
asterisks (`*`) or plus signs (`+`) as bullet markers is strictly prohibited to ensure uniform consistency across all
documentation. Numbered list items must consistently begin with `1.` followed by a single space. "Lazy numbering,"
where `1.` is used for all items and the Markdown renderer automatically increments the numbers, is permitted and
strongly encouraged for ease of maintenance and adaptability. This approach allows AI to infer sequential relationships
without relying on precise human numbering, making documents more robust to editing errors.

For nested lists, whether bulleted or numbered, each successive level must be indented by exactly four (4) spaces from
the starting position of its parent list item's marker. This precise indentation rule applies consistently to both the
list marker and the content of the nested list item, preventing misclassification of nested items and directly
impacting the AI's Coherence Index and Relational Gravity Signature by making parent-child dependencies explicit and
machine-readable. Any subsequent paragraphs, code blocks, or other block-level content belonging to a single list item
must also be indented by four (4) spaces from the list item's marker, separated by a single blank line. This ensures
AI correctly associates all content within a multi-paragraph list item as belonging to that single item, maintaining
semantic coherence.

For lists where all items are concise and fit on a single line, no blank lines should be used between list items.
Conversely, for lists where items are long, contain multiple paragraphs, or include nested elements, a single blank
line must be used between each list item to enhance readability in both the source Markdown and the rendered output.
This nuanced approach balances readability for human authors with parsing efficiency for AI systems.

### 4.4. Code & Technical Content Formatting: Precision for AI Execution

The PGPS mandates precise formatting for code and technical content to ensure optimal AI execution and interpretation.
Inline code snippets, specific artifact IDs, and inline commands must be enclosed in single backticks (`` ` ``) and
integrated directly within the surrounding text without any additional indentation. This allows AI to recognize and
preserve these elements as literal text, preventing unintended parsing or formatting that could alter their meaning,
which is crucial for accurate command interpretation and artifact linking.

Fenced code blocks must consistently begin and end with three backticks (` ``` `) on their own line, with zero
indentation from the left margin. The programming language _must_ be declared immediately after the opening backticks
for syntax highlighting purposes (e.g., `python`). A single blank line must precede and follow the entire code block.
This provides clear, machine-readable boundaries for code, enabling AI to apply specific syntax highlighting,
interpret the content as executable or structured code, and potentially perform static analysis or generate
documentation from it.

For simple, non-language-specific code snippets or legacy content that does not require explicit syntax highlighting,
an indented code block can be created by indenting every line of the code by at least four (4) spaces from the left
margin. While less preferred than fenced code blocks due to the lack of explicit language declaration, consistent
four-space indentation still allows AI to recognize them as distinct code segments, preventing misinterpretation as
regular paragraphs. When a code block (whether fenced or indented) is nested within a list item, its opening and
closing markers (for fenced blocks) or all lines of code (for indented blocks) must be indented by an additional four
(4) spaces beyond the indentation of the list item itself. For a first-level list item, this translates to a total of
eight (8) spaces from the absolute left margin. This rule preserves the logical hierarchy of the list while clearly
delineating the code block as a distinct element belonging to that specific list item, ensuring AI correctly associates
the code with its parent list item and maintains the document's structural and semantic integrity.

### 4.5. Tables: Structured Data Alignment for AI Parsing

The PGPS mandates specific rules for tables to ensure structured data alignment for efficient AI parsing. All tables
must consistently begin and end with a vertical pipe (`|`) on each row, and columns within the table must be separated
by `|`. The header row separator must utilize at least three hyphens (`---`) for each column. A single blank line must
consistently precede and follow the entire table structure. These explicit and unambiguous delimiters enable AI to
accurately identify and parse tabular data, with mandatory leading and trailing pipes eliminating ambiguity in table
boundaries, which is critical for efficient Data Extraction and Relational Gravity Signature analysis.

Column alignment within tables must be explicitly specified using colons (`:`) in the header separator row: a colon on
the left (`:---`) for left-aligned content, a colon on the right (`---:`) for right-aligned content, and colons on
both sides (`:---:`) for center-aligned content. While primarily intended for human readability, consistent alignment
cues can assist advanced AI parsers in understanding the intended presentation and potentially inferring data types or
relationships within table cells. For complex content within table cells, the use of reference-style links is
encouraged to maintain readability and keep cell content concise.
