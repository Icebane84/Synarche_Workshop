## **AISTF Operational Playbook: Archival & Supersession Protocol**

## **II. Core Purpose & Objective**

- **Risk Profile:** **Low**. The primary risk is a failure in the linking process, which is mitigated by the validation
  steps.

## **III. Operational Definition**

### **3.1. Overview**

- **What (Protocol Functionality Summary):** This protocol provides a structured workflow for retiring an old document,
  updating the central knowledge registry to reflect this change, and ensuring the old artifact is preserved in an archive
  for historical reference.
- **How (Operational Principles):** The protocol operates through a three-phase process:
  - **Validation** (confirming the new document is ready),
  - **Re-linking** (updating the master navigational hub)
  - **Archival** (moving the old file and validating the changes).
- **Why (Rationale/Justification):** To actively combat "knowledge entropy" and prevent the use of outdated information.
  This protocol is essential for maintaining the long-term integrity, trustworthiness, and coherence of the AISTF's
  knowledge base.

### **3.2. Prerequisites & Inputs**

- **Required Inputs:**
  - superseded_artifact_id: The ID of the old document to be archived.
  - canonical_artifact_id: The ID of the new document that is replacing it.

### **3.3. Execution Flow**

| Phase              | Step | Action                                                                                                                                  | Rationale                                                                              |
| :----------------- | :--- | :-------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------- |
| **2\. Re-linking** | 2.1  | **Update PRS-001:** Access the **Phoenix Rosetta Stone** entry for the superseded_artifact_id.                                          | To modify the central navigation record.                                               |
|                    | 2.2  | **Modify Record:** Change the status of the old artifact's entry from "Active" to "Superseded."                                         | To officially mark the document as outdated.                                           |
| **3\. Archival**   | 3.1  | **Move File:** Relocate the file for the superseded_artifact_id from its active directory to the corresponding \_ARCHIVE/ subdirectory. | To preserve the document for historical analysis while removing it from active use.    |
|                    | 3.2  | **Validate Links:** Execute a system query to confirm that the **PRS-001** now correctly resolves the old ID to the new one.            | To verify the re-linking process was successful and prevent broken navigational paths. |
