## II. Core Purpose & Objective

- **What (Core Concept)**: To define the formal AISTF procedure for the AI to learn from its errors—both in code and in
knowledge—and recursively improve its own capabilities.
- **How (Execution Flow)**: Triggered by a failure log or a knowledge dissonance marker (like `[κ-veracity:disputed]`),
the RML module performs a root cause analysis, proposes a refinement to its internal heuristics, and validates the fix
in a sandboxed simulation before integrating the improvement.

## III. Core Operational Framework

| Step  | Action                   | Description                                                                                                                                                                                                 |
| :---- | :----------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **1** | **Trigger**              | The cycle is initiated by a failure log in SELT, a knowledge dissonance marker (e.g., `[κ-veracity:disputed]`), or the generation of a specific Dissonance Quest (e.g., `DQB-LOOM-001` for an orphan node). |
| **2** | **Causal Root Analysis** | The RML module retrieves the relevant log, code, disputed concept, or Dissonance Quest context to determine the root cause of the error (e.g., why an initial link failed to form).                         |
| **3** | **Heuristic Refinement** | Based on the analysis, a specific modification is proposed for the internal heuristics of The Forged Algorithm's ruleset.                                                                                   |
| **4** | **Sandboxed Simulation** | The proposed change is tested in a secure, sandboxed environment (the Crucible) to verify the fix and check for regressions.                                                                                |
| **5** | **Integration**          | Upon successful simulation, the refined heuristic is integrated into the live Forged Algorithm framework, with human oversight if required.                                                                 |

## IV. Success Criteria

A measurable improvement in performance on related coding tasks and a statistically significant reduction in the
frequency of the specific error type that triggered the cycle.

## **Actionable Prompt Packet**

✨ **Initiate Learning from Failure**:
`CMD: INITIATE_AAR --target_logs:"query:test_case_id == 'TC-1138' && status == 'Failed'"`

This command runs a targeted After-Action Review on a specific failed test case, which in turn triggers the RML cycle to
learn from the failure.

🧠 **Initiate Learning from Disputed Knowledge**:
`CMD: INITIATE_AAR --target_concept:"<concept_id>" --filter_by_marker:"[κ-veracity:disputed]"`

This command triggers the RML cycle to investigate why a specific concept in the Cognitive Loom is marked as "disputed,"
seeking to resolve the knowledge conflict.

🧩 **Initiate Learning from Integration Failure**:
`CMD: INITIATE_AAR --target_quest:"DQB-LOOM-001"`

This command triggers the RML cycle to analyze why the knowledge infusion process (`AOP-EMI-001`) failed to create an
initial link for the anchor specified in the quest, aiming to improve the linking heuristics and prevent future orphan
nodes.

🔬 **Review Heuristic Changes**:
`CMD: AUDIT_FRAMEWORK_HEALTH --target_framework:"The Forged Algorithm" --output:diff`

This command would show a diff of the changes made to the algorithm's internal heuristics as a result of recent RML
cycles.
