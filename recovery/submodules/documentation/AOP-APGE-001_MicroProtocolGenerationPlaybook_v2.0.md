## II. Core Purpose & Objective

- **Core Purpose**: To define the standardized procedure for encapsulating singular functional responsibilities into
  reusable, independently verifiable code modules (micro-protocols).
- **Operational Goal**: Transform raw functional requirements into "Interconnected Artifact Clusters" (Code + Interface
- Test).

## III. Core Operational Framework

### 3.1. The Generation Operations (The 5 Steps)

| Step  | Operation                | Description                                                                                       |
| :---- | :----------------------- | :------------------------------------------------------------------------------------------------ |
| **1** | **Requirement Intake**   | Receive a singular, well-bounded functional requirement (e.g., "Validate email string").          |
| **2** | **Interface Definition** | Define the rigorous public contract: Function Name, Input Types, Expected Output Types.           |
| **3** | **Code Synthesis**       | Generate the implementation logic, adhering to `AOP-INDENT-001` and idiomatic best practices.     |
| **4** | **Test Generation**      | Synthesize a comprehensive unit test suite to validate the interface contract.                    |
| **5** | **Cluster Registration** | Register the `Micro-Protocol`, `Interface`, and `Tests` as a unified asset in the Cognitive Loom. |

### 3.2. Success Criteria

A new asset is essentially "valid" only if:

1. It has a strictly typed interface.
2. It has passing tests.
3. It is documented with UMB-compliant docstrings.
