## II. Core Purpose & Objective

- **What (Core Concept):** A governance protocol that defines the complete, secure lifecycle for an external data source
  ("Oracle"), from initial nomination and vetting to active use, ongoing maintenance, and eventual deprecation.
- **How (Execution Flow):** The protocol is a multi-phase process initiated by the AI but gated by mandatory human
  authorization. It involves rigorous vetting, secure credential management via a dedicated vault, and continuous
  performance monitoring.

## III. Protocol Lifecycle & Phases

The lifecycle of an Oracle is managed through four distinct, sequential phases.

| Phase   | Title                       | Description                                                                                                                                                                                                                                                               |
| :------ | :-------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **I**   | **Nomination & Discovery**  | The `Coherent Synthesis Engine (CSE)` identifies a potential new data source that could enhance its knowledge base. It generates a "Candidate Profile" detailing the source's scope, endpoint, and expected data type.                                                    |
| **II**  | **Vetting & Authorization** | A **mandatory human-in-the-loop security gate**. The Candidate Profile is presented to the `Human User` via the `Rosetta Stone App`. The user must explicitly "Authorize" or "Reject" the candidate. This is the only path to promotion. [cite: `AOP_ORACLE_VETTING_001`] |
| **III** | **Secure Registration**     | Upon authorization, the Oracle is formally added to the `Oracle Registry`. Its credentials (API keys, tokens) are **never** stored in the registry itself but are deposited into a `Secure Vault`. The registry only stores a non-sensitive pointer to the secret.        |
| **IV**  | **Maintenance & Review**    | The `Truth Resonance Monitor (TRM)` continuously queries the active Oracle and updates its `Reliability Score` in the registry. If the score falls below a set threshold, the Oracle is automatically quarantined and flagged for human review.                           |
