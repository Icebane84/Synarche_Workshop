   transactional event.
2. **Payload Transmission:** The Phoenix Protocol Library will immediately fire a lightweight HTTPS payload (a webhook)
   containing the unique **Entity ID** (e.g., `service_id_456`, `packet_id_007`), the **New State Value**, and the
   **Source Data Field** name.
3. **Middleware Component:** This payload is received by a **lightweight Middleware Component (LMC)**, which serves as
   the primary interface between the two systems. The LMC is stateless and highly performant.
4. **`PHOENIX-EMOJICXT-001` Query:** The LMC uses the **Source Data Field** and the **New State Value** to query the
   `PHOENIX-EMOJICXT-001` service API.
5. **Emoji Retrieval:** `PHOENIX-EMOJICXT-001` processes the request against its dynamic mapping tables and returns the
   relevant **Emoji URI** (for image-based rendering in older clients) or the **Unicode Character/String** (for native
   rendering).
6. **Dynamic Rendering:** The LMC passes the retrieved emoji data back to the Phoenix Protocol Library UI component
   (e.g., the web console). The Library then dynamically renders the emoji in the appropriate context (e.g., next to the
   status message, in the log entry, or as the icon for a service section).
